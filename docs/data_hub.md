# Codex Implementation Brief: Data Hub / Data Lake Module for ILM

## Goal

Add a new **Data Hub** module to the Integrated Lab Manager (ILM).

The module should help a lab track, discover, request, and reuse datasets generated across projects or obtained from external sources. It should **not** try to store large data files in the app. Instead, it should act as a **dataset registry + reuse/access workflow + lightweight lineage tracker**.

Use **Data Hub** as the user-facing module name. Use **Data Lake** as the conceptual backend/storage metaphor where helpful, but do not imply that ILM itself stores raw datasets.

Core question the module should answer:

> What datasets do we have, where are they, who owns them, what are they for, can I use them, and how have they been reused across projects?

---

## Product Philosophy

Treat datasets as first-class lab assets.

A dataset is not just a folder path. It is a reusable scientific object with:

1. Metadata
2. Storage location
3. Owner/contact
4. Access rules
5. Linked projects
6. Version/processing history
7. Reuse/request history
8. Notes about limitations and recommended usage

The app should prioritize **discoverability and reuse** over file management.

Important design rule:

> By default, datasets should be discoverable, but not necessarily downloadable.

---

## MVP Scope

Implement a practical v1 module with the following features:

1. Create, view, edit, archive datasets
2. Search/filter datasets in a Data Hub library
3. Link datasets to one or more projects
4. Track dataset owner/contact/status/access level
5. Store dataset location as path/link/accession, not file upload
6. Add simple tags
7. Add dataset versions and basic lineage
8. Allow users to submit dataset access/reuse requests
9. Allow dataset owners/admins to approve, deny, or withdraw requests
10. Show dataset detail pages with summary, storage, projects, versions, requests, and notes

Do not implement actual file transfer, cloud synchronization, checksum indexing, or automated HPC scanning in v1.

## Supabase/Postgres Schema

Adapt field names to the existing style if needed.

```sql
create table if not exists datasets (
  id uuid primary key default gen_random_uuid(),

  name text not null,
  description text,
  dataset_type text not null default 'other',
  source_type text not null default 'internal-generated',
  status text not null default 'planned',
  access_level text not null default 'request-required',

  owner_user_id uuid,
  contact_user_id uuid,

  organism text,
  sample_type text,
  assay_platform text,

  primary_storage_uri text,
  external_accession text,
  citation text,
  license text,
  usage_conditions text,

  recommended_use text,
  not_recommended_use text,
  qc_summary text,
  notes text,

  created_by uuid,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  archived_at timestamptz
);

create table if not exists dataset_versions (
  id uuid primary key default gen_random_uuid(),

  dataset_id uuid not null references datasets(id) on delete cascade,
  version_name text not null,
  version_type text not null default 'processed',
  description text,

  storage_uri text,
  parent_version_id uuid references dataset_versions(id) on delete set null,

  processing_summary text,
  software_environment text,
  qc_summary text,
  file_summary text,
  notes text,

  created_by uuid,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists dataset_project_links (
  id uuid primary key default gen_random_uuid(),

  dataset_id uuid not null references datasets(id) on delete cascade,
  project_id uuid not null,
  relationship_type text not null default 'used-by',
  note text,

  created_by uuid,
  created_at timestamptz not null default now(),

  unique(dataset_id, project_id, relationship_type)
);

create table if not exists dataset_access_requests (
  id uuid primary key default gen_random_uuid(),

  dataset_id uuid not null references datasets(id) on delete cascade,
  dataset_version_id uuid references dataset_versions(id) on delete set null,

  requester_user_id uuid,
  project_id uuid,
  intended_use text not null,
  requested_access_type text not null default 'view-and-use',

  status text not null default 'pending',

  reviewer_user_id uuid,
  decision_note text,
  conditions text,

  created_at timestamptz not null default now(),
  reviewed_at timestamptz,
  withdrawn_at timestamptz
);

create table if not exists dataset_tags (
  id uuid primary key default gen_random_uuid(),

  dataset_id uuid not null references datasets(id) on delete cascade,
  tag text not null,

  created_at timestamptz not null default now(),

  unique(dataset_id, tag)
);

create table if not exists dataset_storage_links (
  id uuid primary key default gen_random_uuid(),

  dataset_id uuid not null references datasets(id) on delete cascade,
  dataset_version_id uuid references dataset_versions(id) on delete cascade,

  label text,
  storage_uri text not null,
  storage_type text not null default 'path',
  notes text,

  created_by uuid,
  created_at timestamptz not null default now()
);
```

Notes:

- `project_id` should reference the existing projects table once its exact table name/id type is known.
- `owner_user_id`, `contact_user_id`, `created_by`, `requester_user_id`, and `reviewer_user_id` should reference the existing users/profiles/auth table if present.
- If the current schema uses integer IDs instead of UUIDs, adapt consistently.
- Add indexes on `dataset_id`, `project_id`, `owner_user_id`, `status`, `access_level`, and `tag`.

Recommended indexes:

```sql
create index if not exists idx_datasets_owner on datasets(owner_user_id);
create index if not exists idx_datasets_status on datasets(status);
create index if not exists idx_datasets_access_level on datasets(access_level);
create index if not exists idx_dataset_versions_dataset on dataset_versions(dataset_id);
create index if not exists idx_dataset_project_links_dataset on dataset_project_links(dataset_id);
create index if not exists idx_dataset_project_links_project on dataset_project_links(project_id);
create index if not exists idx_dataset_access_requests_dataset on dataset_access_requests(dataset_id);
create index if not exists idx_dataset_access_requests_requester on dataset_access_requests(requester_user_id);
create index if not exists idx_dataset_access_requests_status on dataset_access_requests(status);
create index if not exists idx_dataset_tags_dataset on dataset_tags(dataset_id);
create index if not exists idx_dataset_tags_tag on dataset_tags(tag);
```

## Row-Level Security / Permissions

If Supabase RLS is already used, add policies consistent with the rest of ILM.

Recommended logic:

### Dataset visibility

- `open-lab`: visible to all authenticated lab users
- `request-required`: metadata visible to all authenticated lab users, storage details may be hidden unless owner/admin/approved requester
- `restricted`: visible to owner/admin and explicitly approved users only
- `private`: visible only to owner/admin

For v1, it is acceptable to implement this logic in the frontend/service layer first, but design the schema so RLS can be added.

### Request permissions

- Any authenticated user can create an access request for a visible dataset.
- Requester can view their own requests.
- Dataset owner/contact/admin can view and review requests for their datasets.
- Approved requests should create or expose a dataset-project relationship if appropriate.

## Acceptance Criteria

The implementation is complete when:

1. A user can create a dataset record with metadata and storage location.
2. A user can browse/search/filter datasets in Data Hub.
3. A dataset can be linked to multiple projects.
4. A dataset can have multiple versions.
5. A version can point to a parent version.
6. A user can request access/reuse of a dataset.
7. A dataset owner/admin can approve or deny a request.
8. Approved reuse is visible on the dataset detail page.
9. External datasets can be registered with accession/citation/license information.
10. The UI makes clear that ILM stores metadata and locations, not raw large files.
11. Existing ILM modules continue to work.

## Naming Decision

Use:

```text
User-facing module: Data Hub
Conceptual backend/storage metaphor: Data Lake
Technical object: Dataset Registry
```

Rationale:

- **Data Hub** feels approachable and product-friendly.
- **Data Lake** captures the idea of scattered data storage, but should not imply ILM stores all raw files.
- **Dataset Registry** is the precise technical implementation.
